# ROUTING PHILOSOPHY — AI_ASSISTANT_CORE

## Principe fondamental
Une demande ne doit pas produire un comportement implicite ou caché.

Le système doit rendre visible :
- la tâche reconnue
- la règle appliquée
- le tool retenu
- le plan construit
- la sortie finale utile
- les erreurs runtime importantes

## Hiérarchie actuelle
1. `task_classifier`
2. `TASK_ROUTING`
3. `routing_conditions`
4. `router_service`
5. `plan_builder`
6. `step_executor`
7. `result_assembler`

## Rôle des composants
- classifier : signal initial, pas vérité absolue
- `TASK_ROUTING` : source statique principale
- `routing_conditions` : variation limitée et lisible
- `router_service` : autorité de décision finale
- `executor` : exécute, trace et assemble ; il ne recrée pas une logique métier cachée

## Place de la couche visuelle session 4
L’analyse `subject_type / render_intent / style_flags` n’est pas un second routeur général.
C’est une couche spécialisée du sous-pipeline visuel destinée à mieux préparer ComfyUI.

## Exception assumée
`image_generation` force actuellement `selected_tool = comfyui` dans `router_service`.
C’est une exception explicite, visible et traçable.

## Discipline
- les règles hybrides doivent rester courtes et testables
- la préparation visuelle doit rester lisible et déterministe
- les forced modes côté OpenAI-compatible doivent rester contrôlés
- la surface API ne doit pas mentir sur ce qui est exécuté

## Anti-patterns à éviter
- logique métier cachée dans l’executor
- duplication entre racine et `app/*`
- docs de version en retard
- nouvelle surcouche externe pour masquer de l’instabilité interne
- requalifier comme “dormant” un module réellement appelé par le pipeline visuel
